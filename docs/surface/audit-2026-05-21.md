Date: 2026-05-21

## Detected Stack

- TypeScript pnpm/turbo monorepo. Root scripts cover build, lint, typecheck, tests, and `pnpm check` in `package.json:4-16`; package manager is `pnpm@10.23.0` in `package.json:36`.

- No HTTP API, hosted web app, public OpenAPI spec, remote MCP transport, retrieval/RAG corpus, or multi-agent runtime in this public repo.

## Findings

### 1. Creative Direction Is Now A Shared Agent Surface

The imported photography taxonomy has been converted into a reusable, typed SDK-owned contract and exposed through the major agent entrypoints.

| Severity | File | Status | Dimension |
| --- | --- | --- | --- |
| Resolved | `packages/motif-sdk/src/creative.ts` | SDK owns creative fields, options, clauses, prompt enrichment, and structured invalid-option errors. | Tool Design |
| Resolved | `apps/cli/src/commands/describe.ts` | `motif --describe generate --format json` exposes creative enum ids with labels, descriptions, and clause previews. | CLI Design, Tool Design |
| Resolved | `packages/motif-mcp/src/create-server.ts` | MCP `generate` and `vary` expose and pass through the same creative object. | MCP Server, Tool Design |
| Resolved | `apps/cli/tests/cli-contract.test.ts`, `packages/motif-mcp/tests/tools.test.ts` | Contract tests cover creative schema, dry-run metadata, Series prompt enrichment, and MCP pass-through. | Testing |

Suggested approach: keep adding taxonomy entries in the SDK only, then let CLI/MCP schemas render from that source.

Dependencies: none.

### 2. CLI Error Semantics Still Stop Short Of Full Autonomy

The CLI emits useful structured error objects, but process-level semantics and trace correlation are still not rich enough for fully autonomous recovery.

| Severity | File | Issue | Dimension |
| --- | --- | --- | --- |
| Medium | `apps/cli/src/utils/errors.ts`, `apps/cli/src/utils/output.ts` | Structured error body includes status/doc_uri/suggestions, but process exit codes are not semantic. | CLI Design, Error Handling |
| Medium | `apps/cli/src/utils/output.ts` | No `trace_id` or `instance` field for correlating failed CLI runs. | Error Handling |
| Low | `apps/cli/src/commands/describe.ts` | Error docs are local schema entries, not stable markdown pages. | Error Handling, Discovery |

Suggested approach: map error catalog metadata to sysexits-style exit codes, add generated `trace_id`/`instance` to structured errors, and document error code pages under `docs/errors/`.

Dependencies: decide whether changing exit codes is a breaking CLI contract.

### 3. MCP Is Agent-First For Local Stdio, Not A Remote Protected Server

The local stdio MCP surface is strong, but there is no remote transport, OAuth protected-resource metadata, or long-running task/progress model.

| Severity | File | Issue | Dimension |
| --- | --- | --- | --- |
| Resolved | `packages/motif-mcp/src/create-server.ts` | Tools have descriptions, annotations, output schemas, structured content, resources, and structured recoverable errors. | MCP Server |
| Low | `packages/motif-mcp/src/index.ts` | Startup failure for missing `FAL_KEY` is a plain fatal message. | Authentication, Error Handling |
| Low | repo root | No Streamable HTTP transport or RFC 9728 metadata, which is acceptable for local-only stdio but would be required if hosted. | MCP Server, Authentication |

Suggested approach: keep the local stdio boundary explicit. Only add HTTP/OAuth metadata if Motif becomes a hosted MCP server.

Dependencies: future hosting decision.

### 4. Discovery Is Good For A Public Repo, Not A Full Web Surface

Motif has strong repo/package discovery, but not the hosted web signals expected from a documentation site.

| Severity | File | Issue | Dimension |
| --- | --- | --- | --- |
| Resolved | `AGENTS.md`, `CLAUDE.md`, `llms.txt`, `.mcp.json` | Repo-local and local MCP discovery are in place. | Discovery, Context Files |
| Low | repo root | No `llms-full.txt`, `.well-known` API/MCP cards, robots policy, sitemap, or Markdown content negotiation. | Discovery |

Suggested approach: do not add web-only machinery unless Motif gets hosted docs. If docs are published, generate `llms-full.txt`, `.well-known` capability cards, sitemap, and Markdown fallbacks.

Dependencies: hosted docs/site decision.

### 5. Testing Is Strong Deterministically, Missing Agent Eval Fixtures

The deterministic test suite covers the CLI, SDK, MCP protocol, and package smoke paths, but there is no labeled agent-task regression suite.

| Severity | File | Issue | Dimension |
| --- | --- | --- | --- |
| Resolved | `packages/motif-mcp/tests/tools.test.ts` | MCP protocol tests cover annotations, resources, structured errors, and creative pass-through. | MCP Server, Testing |
| Resolved | `apps/cli/tests/cli-contract.test.ts` | CLI contract tests cover describe schema, dry runs, stdin JSON, structured errors, utility tools, and Series creative prompts. | CLI Design, Testing |
| Medium | repo root | No `tests/agent-tasks/*.jsonl` fixture suite for recurring CLI/MCP agent-task regression. | Testing |
| Low | `.github/workflows/ci.yml` | CI runs deterministic tests but no optional eval/canary reporting job. | Testing |

Suggested approach: add a small fixture set of representative agent tasks and expected structured outputs, all using dry-run CLI calls or mocked MCP calls. Keep live fal canaries opt-in.

Dependencies: none.
