# Motif 1.0 — Clarity Reshape (design for review)

Status: PROPOSED — design under review before implementation. Supersedes the rejected
"1d fold" (see 1d-motifserver-fold-approach.md). MOT-24 → 1.0.

## Why (the problem, stated plainly)

The additive 1a-1c work shipped a provider-agnostic image layer alongside the original
fal client. That was the right way to *build* it, but it left the package **confusing**:

1. **Two ways to generate an image** — `new MotifServer().generate()` (fal only) and
   `createMotifImage().generate()` (4 providers). "How do I generate?" has two answers.
2. **`MotifServer` is a generically-named, fal-specific grab-bag** — named "Motif" but
   fal-only, mixing generation with upscale/rmbg/video/tools/queue/upload.
3. **fal generation is implemented twice** — `MotifServer.generate` (native `fal.run` +
   queue lifecycle) and `src/image/fal.ts` (via `@ai-sdk/fal`). Two codepaths, one job.

This is NOT the rejected 1d fold. We are NOT forcing fal-only operations (upscale, rmbg,
video, tools) into a provider-agnostic interface — the review correctly killed that. We
ARE renaming/repositioning for one-obvious-way clarity, and de-duplicating fal generation.

## Consumer reality (why 1.0 is cheap here)

Only ONE external consumer exists: `~/Sites/kiln/packages/generation`, pinned to
`@howells/motif-sdk@^0.2.0` (locked to 0.2.x — our 0.7-0.10 work never reached it). It
uses `MotifServer` + `MODELS` + `GenerateOptions` + `.generate()` with retry backoff. A
breaking 1.0 does not touch kiln until someone deliberately bumps it; a one-page migration
note covers it. CLI + MCP are in-repo and we migrate them ourselves. So the usual "don't
break published consumers" cost is near-zero.

## Target shape (one obvious way to do each thing)

`@howells/motif-sdk` exposes two clearly-separated surfaces:

- **Primary — provider-agnostic image generation** (`@howells/motif-sdk/image`,
  `createMotifImage`): `generate` / `edit` / `bestOfN` across google/openai/replicate/fal.
  This is THE documented way to generate and edit images. Promote it from "subpath" to the
  headline of the package.
- **fal specialist client — renamed `MotifServer` → `FalClient`** (core `.` entry): the
  fal-only capabilities that are genuinely single-provider — `upscale`, `removeBackground`,
  `video`, `runTool` (fal utilities), `uploadToFalCdn`, `deletePayloads`, and the fal queue
  lifecycle. Named so it's unmistakable this is fal-specific, not "the Motif client."

`MODELS` and model metadata stay as shared exports.

## The decisive question: the two fal generation paths (dedupe)

`FalClient.generate` handles fal **queue-required** models (`MODELS.gptImage2.useQueue`,
submit→poll→result). The image layer's fal adapter uses `@ai-sdk/fal`'s synchronous
`generateImage`.

**SPIKE RESOLVED (2026-07-13, live):** `@ai-sdk/fal` handles fal's queue-backed models
**transparently** — a single `generateImage` call against `fal-ai/gpt-image-1.5` (a
queue-required model) resolved in ~35s (matching the native path's benchmark), so it polls
the queue under the hood. **→ YES branch:** the image layer's fal adapter fully subsumes
`FalClient.generate` and the queue lifecycle. Fal generation lives in ONE place (the image
layer). `FalClient` drops `generate`/`submitGeneration`/`getJobStatus`/`getJobResult`
entirely and becomes purely the fal-specific extras (upscale, removeBackground, video,
runTool, uploadToFalCdn, deletePayloads).

User-facing rule: **generate/edit via `createMotifImage`; reach for `FalClient` only for
fal-specific extras.**

## Migration

- **kiln**: `MotifServer` → `FalClient` (drop-in if we keep `generate` per the spike's NO
  branch), or migrate its generation to `createMotifImage` (YES branch). One-page note;
  not forced (it's pinned to 0.2.x).
- **CLI** (`apps/cli/src/api/fal.ts` + generate/postprocess/video/tools commands): route
  generation through `createMotifImage`; route upscale/rmbg/video/tools through `FalClient`.
- **MCP** (`create-server.ts`): same split — generate/vary via image layer, upscale/rmbg
  via `FalClient`.
- **`packages/motif-server`**: already a deprecated re-export wrapper — retire it at 1.0.

## Phasing

- **A — spike**: resolve the `@ai-sdk/fal` queue question (one cheap probe). Decides dedupe.
- **B — rename**: `MotifServer` → `FalClient` (core), keep a deprecated `MotifServer` alias
  export for one minor so the rename itself isn't a hard break mid-flight.
- **C — dedupe generation** per the spike result.
- **D — migrate CLI + MCP** to the two-surface model; update AGENTS.md Public Surface.
- **E — docs + kiln note**; retire `packages/motif-server`; cut 1.0; remove the alias.

## Non-goals

- NOT folding upscale/rmbg/video/tools into a provider-agnostic interface (rejected 1d).
- NOT inventing multi-provider contracts for single-provider (fal) operations.
- NOT changing the image layer's proven 1a-1c API shape — only promoting it to primary.

## Open questions

1. The `@ai-sdk/fal` queue spike (blocks the dedupe decision).
2. `FalClient` naming — `FalClient` vs `FalImageClient` vs keeping `MotifServer` as an
   alias permanently. (Recommend `FalClient`, alias `MotifServer` for one deprecation minor.)
3. Does CLI's terminal Studio rely on any `MotifServer` behavior that doesn't cleanly split?
4. Version: is this 1.0, or 0.11 additive-then-1.0? (Recommend: do the rename additively
   with an alias in 0.11, cut 1.0 when the alias is removed.)
